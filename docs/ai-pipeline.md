## 2. 현재 구현 체크포인트 (2026-04-08)

본 문서는 이상적인 파이프라인 설계를 포함하지만, 현재 저장소 기준으로는 아래 범위를 우선 구현했습니다.

- 자료 처리: `uploaded -> queued -> processing -> completed/failed` 상태 전이
- 자료 파이프라인: `app/pipelines/pdf_orchestrator.py`에서 단계 함수 호출 순서를 한 눈에 확인 가능
- PDF 입력 수집: `app/services/storage_service.py`에서 `fileUrl` 기준 파일 바이트 다운로드(직접 GET + Supabase service role fallback)
- PDF 신호 추출: `app/parsers/pdf_parser.py`에서 PyMuPDF 기반 페이지 텍스트/신호 추출
- 녹화본 처리: `uploaded -> queued -> transcribing -> post_processing -> completed/failed` 상태 전이
- STT 처리: `app/services/stt_service.py`에서 OpenAI STT API 호출(실패 시 데모 세그먼트 fallback)
- 자막 후보정: `app/services/llm_service.py`에서 세그먼트 단위 텍스트 보정(키 미설정 시 원문 유지)
- 퀴즈 작업: `queued -> processing -> completed/failed` 상태 전이
- 워커 구조: PostgreSQL(`kit_contest`) DB 폴링(기본 5초) + 배치 처리
- 공통 데모 콘텐츠 생성 로직: 프론트/워커 공통 유틸로 분리

현재 구현은 데모용 안정 경로 확보를 목표로 하며, 외부 STT/LLM 실연동은 다음 단계에서 확장합니다.

### 2-1. PDF 오케스트레이터 구조(현재 구현)

현재 워커의 PDF 처리 경로는 다음 구조를 따른다.

- `jobs/material_job.py`: 처리 대상 material 조회와 배치 루프 담당
- `pipelines/pdf_orchestrator.py`: 단계 호출 순서 오케스트레이션 담당
- `pipelines/pdf_pipeline.py`: 각 단계의 세부 처리 함수 담당

오케스트레이터는 아래와 같이 “함수 호출만으로” 흐름을 표현한다.

1. 상태 전이(`uploaded -> queued -> processing`)
2. 입력 소스 정규화
3. 페이지 구조 신호 추출
4. 전역 맥락 요약
5. 목차 후보/확정
6. 페이지-목차 매핑
7. 섹션/페이지 저장
8. 완료 상태 업데이트(`completed`)

실패 시 공통 실패 처리 함수로 상태를 `failed`로 전환한다.

### 2-2. STT/퀴즈 오케스트레이터 구조(현재 구현)

오케스트레이터 패턴을 STT/퀴즈에도 동일하게 적용한다.

- STT: `jobs/recording_job.py -> pipelines/stt_orchestrator.py -> pipelines/stt_pipeline.py`
- 퀴즈: `jobs/quiz_job.py -> pipelines/quiz_orchestrator.py -> pipelines/quiz_pipeline.py`

각 오케스트레이터는 상태 전이, 핵심 단계 호출, 완료/실패 반영만 담당하고, 세부 로직은 파이프라인 함수에서 캡슐화한다.

STT의 경우 `ensure_segments` 단계에서 다음 순서로 동작한다.

1. 스토리지에서 녹화본 파일 바이트 다운로드
2. OpenAI STT API로 타임스탬프 세그먼트 추출
3. 자료 목차와 시간축 기반으로 섹션/페이지 매핑
4. LLM 후보정으로 `refinedText` 생성
5. 실패 시 데모 세그먼트 생성 fallback

## 3. PDF 강의자료 구조화 파이프라인

### 목적
강의자료 PDF를 단순 파일이 아니라,  
목차/페이지/핵심 개념 단위로 탐색 가능한 학습 구조로 변환한다.

이 파이프라인은 공모전 제출용 라이브 웹 서비스에서 실제로 동작 가능한 형태를 목표로 하며,  
업로드 직후 즉시 분석을 수행하는 것이 아니라 상태 기반 비동기 처리 방식으로 동작한다.

---

### 입력
- PDF 파일
- 세션 정보
- 자료 유형
- 정보 추출 활성화 여부

---

### 처리 단계

#### 0) 파일 검증
- 파일 형식이 PDF가 맞는지
- 파일 크기가 제한을 초과하지 않는지
- 텍스트 추출이 가능한 정상 PDF인지

#### 1) 파일 저장
- 업로드된 PDF를 스토리지에 저장한다.
- `materials` 레코드를 생성한다.
- 초기 상태를 `uploaded`로 저장한다.

#### 2) 정보 추출 여부 확인
- `extract_enabled = false` 인 경우:
  - 파일 업로드만 완료한다.
  - 추가 AI 처리 없이 종료한다.
- `extract_enabled = true` 인 경우:
  - 상태를 `queued`로 업데이트한다.
  - 자료를 대기 큐에 등록한다.

#### 3) 대기 큐 기반 순차 처리
- 대기 큐에는 `queued` 상태의 자료만 들어간다.
- MVP 기준의 대기 큐는 외부 메시지 브로커가 아니라 DB 상태값 기반 큐를 의미한다.
- 워커 또는 배치 프로세스는 대기 큐의 자료를 순차적으로 가져와 처리한다.
- 하나의 자료 처리가 시작되면 상태를 `processing`으로 변경한다.
- 처리 중 오류가 발생하면 상태를 `failed`로 변경하고 종료한다.
- 모든 처리가 정상 종료되면 상태를 `completed`로 변경한다.

---

### 상세 분석 단계

PDF 구조화는 한 번에 전체 텍스트를 LLM에 넣어 목차를 생성하는 방식이 아니라,  
**전역 구조를 먼저 가볍게 추정한 뒤, 세부 페이지 분석으로 내려가는 다단계 방식**으로 수행한다.

---

#### 4) 페이지별 텍스트 추출
- PDF 파서를 통해 페이지 단위로 텍스트를 추출한다.
- 각 페이지별로 원본 텍스트 길이, 텍스트 존재 여부, 추출 품질을 기록한다.
- 텍스트가 거의 없거나 비정상적으로 짧은 경우는 이미지 기반 페이지 또는 추출 실패 후보로 표시한다.
- 필요 시 OCR 대상 후보로 분류할 수 있다.

**저장 예시**
- page_number
- raw_text
- text_length
- extraction_quality

---

#### 5) 페이지 구조 신호 추출
목차를 바로 만들지 않고, 먼저 각 페이지에서 **구조 판단에 필요한 신호**를 추출한다.

추출 대상 예시:
- 페이지 상단 제목 후보
- 제목처럼 보이는 짧은 문장
- 번호 패턴 (`1.`, `1.1`, `Chapter 2`, `Part 3` 등)
- 반복적으로 등장하는 헤더/푸터 제거 결과
- 핵심 명사/키워드 후보
- 페이지 유형 추정
  - 표지
  - 목차 페이지
  - 본문 설명 페이지
  - 예제/실습 페이지
  - 정리/요약 페이지
  - 참고자료 페이지

이 단계의 목적은 각 페이지 전체 내용을 모두 쓰는 것이 아니라,  
문서 전체 흐름을 잡기 위한 **요약된 구조 정보(page signature)** 를 만드는 것이다.

**저장 예시**
- title_candidate
- heading_pattern
- page_type
- keyword_candidates
- cleaned_text_preview

---

#### 6) 문서 전역 맥락 요약 생성
페이지 구조 신호들을 모아 문서 전체를 대표하는 **전역 요약 정보**를 생성한다.

이 단계에서는 모든 페이지의 전체 텍스트를 사용하는 대신,  
각 페이지에서 추출한 구조 신호를 기반으로 문서의 큰 주제와 흐름을 추정한다.

생성 대상 예시:
- 문서의 전체 주제
- 주요 단원 후보
- 문서 진행 순서의 큰 흐름
- 반복적으로 등장하는 핵심 개념

이 단계는 이후 목차 생성과 페이지 주제문 생성 시  
**전역 컨텍스트로 참조되는 기준 정보**가 된다.

**핵심 목적**
- 각 페이지를 독립적으로 보지 않도록 전역 맥락을 먼저 만든다.
- 모든 페이지 텍스트를 그대로 LLM에 넣는 비효율을 줄인다.

---

#### 7) 목차 후보 추출
전역 맥락 요약 + 페이지 구조 신호를 바탕으로 목차 후보를 생성한다.

우선순위는 다음과 같다.

1. PDF 내부 목차 정보가 있으면 우선 사용
2. 없으면 제목 후보, 번호 패턴, 페이지 흐름을 바탕으로 규칙 기반 목차 후보 생성
3. 필요 시 LLM을 사용해 후보를 정리하고 계층 구조를 보정

이 단계에서는 목차를 새로 창작하기보다,  
페이지 구조 신호를 기반으로 **문서에 실제 존재하는 구조를 복원**하는 방향으로 처리한다.

생성 대상:
- 대목차
- 소목차
- 각 목차의 시작 페이지
- 각 목차의 종료 페이지 후보

---

#### 8) 목차 정제 및 페이지 범위 확정
생성된 목차 후보를 정제하여 최종 목차 구조를 확정한다.

정제 기준 예시:
- 지나치게 짧은 구간 병합
- 제목 중복 제거
- 앞뒤 페이지 흐름과 맞지 않는 목차 제거
- 제목 수준(level) 정리
- 섹션 간 페이지 범위 충돌 해소

이 단계 결과로 `material_sections`에 저장할 수 있는 구조를 만든다.

---

#### 9) 페이지-목차 매핑
확정된 목차를 기준으로 각 페이지가 어느 섹션에 속하는지 매핑한다.

매핑 기준 예시:
- 목차 시작 페이지 기준
- 제목 후보 일치도
- 인접 페이지와의 내용 연속성
- 키워드 유사도

이 단계가 완료되면 각 페이지는 단순 PDF 페이지가 아니라  
특정 섹션에 소속된 학습 단위가 된다.

---

#### 10) 페이지별 주제문/요약/키워드 생성
이 단계에서는 **확정된 전역 맥락과 페이지가 속한 목차 정보**를 함께 사용하여  
페이지별 분석 결과를 생성한다.

즉, 페이지 단독으로 해석하지 않고 다음 정보를 함께 참고한다.
- 문서 전체 주제
- 상위 목차명
- 인접 페이지 흐름
- 해당 페이지 원문

생성 대상:
- 페이지 주제문
- 짧은 요약
- 핵심 키워드

이 구조를 통해,  
페이지별 분석이 전체 강의 흐름과 어긋나지 않도록 한다.

---

#### 11) 임베딩 생성 및 검색 인덱싱
질의응답과 검색 기능을 위해 텍스트를 임베딩한다.

임베딩 대상:
- 페이지별 요약 텍스트
- 페이지 원문 청크
- 목차 제목 및 설명

필요 시 다음 기준으로 청킹한다.
- 페이지 단위
- 목차 단위
- 페이지 내 문단 단위

생성된 임베딩은 벡터 저장소에 저장한다.

---

#### 12) 결과 저장
다음 결과를 저장한다.

- `materials`
  - processing_status = `completed`
- `material_sections`
  - 확정된 목차 구조
- `material_pages`
  - 페이지별 주제문, 요약, 키워드, 매핑 정보
- 벡터 저장소
  - 질의응답용 임베딩 데이터

---

### 출력
- 문서 전역 주제
- 목차 구조
- 페이지-목차 매핑 정보
- 페이지별 주제문/요약/키워드
- 검색 가능한 임베딩 데이터

---

### 설계 의도

본 파이프라인은 다음 문제를 해결하기 위해 설계한다.

#### 문제 1. 모든 페이지 텍스트를 한 번에 LLM에 넣는 것은 비효율적이다.
- 해결: 페이지 구조 신호를 먼저 추출하고, 그 결과를 이용해 전역 구조를 요약한다.

#### 문제 2. 전역 맥락 없이 페이지별 주제문을 생성하면 품질이 흔들릴 수 있다.
- 해결: 문서 전체 주제와 목차를 먼저 확정한 뒤 페이지별 분석을 수행한다.

#### 문제 3. 목차를 지나치게 생성형으로 만들면 실제 문서 구조와 어긋날 수 있다.
- 해결: PDF 내부 정보, 규칙 기반 후보, LLM 보정을 결합하여 복원형 방식으로 목차를 생성한다.


## 4. 녹화본 자막(STT) 파이프라인

### 목적
강의 녹화본을 단순 영상 파일이 아니라,  
자막/타임스탬프/목차 단위로 탐색 가능한 학습 데이터로 변환한다.

이 파이프라인은 공모전 제출용 라이브 웹 서비스에서 실제로 동작 가능한 형태를 목표로 하며,  
업로드 직후 즉시 분석을 수행하는 것이 아니라 상태 기반 비동기 처리 방식으로 동작한다.

---

### 입력
- 녹화본 파일
- 세션 정보
- 연결된 강의자료 정보
- STT 활성화 여부
- 원본 언어 정보(선택)
- 자막 기본 언어 정보(선택)

---

### 모델 선택 전략

#### 기본 전략
- 기본 STT 모델은 `gpt-4o-transcribe`를 사용한다.
- 비용/속도 우선 시 `gpt-4o-mini-transcribe`를 대안으로 사용한다.
- 기본 응답 형식은 `verbose_json`으로 설정한다.
- 기본 타임스탬프 단위는 `segment`로 사용한다.
- `word` 타임스탬프는 디버깅, 정밀 자막 편집, 단어 단위 하이라이트가 필요할 때만 선택적으로 사용한다.

#### 조건부 전략
- Q&A, 토론, 패널형 수업처럼 다중 화자 구분이 중요한 경우에는
  `gpt-4o-transcribe-diarize`를 사용한다.
- 이 경우 speaker label과 segment timestamp를 함께 활용한다.

---

### 처리 단계

#### 0) 파일 검증
- 파일 형식이 지원되는 영상/오디오 포맷인지 확인한다.
- 파일 크기가 시스템 제한을 초과하지 않는지 확인한다.
- 재생 가능한 정상 파일인지 확인한다.

#### 1) 파일 저장
- 업로드된 녹화본을 스토리지에 저장한다.
- `recordings` 레코드를 생성한다.
- 초기 상태를 `uploaded`로 저장한다.

#### 2) STT 활성화 여부 확인
- STT 비활성화인 경우:
  - 파일 업로드만 완료한다.
  - 추가 AI 처리 없이 종료한다.
- STT 활성화인 경우:
  - 상태를 `queued`로 업데이트한다.
  - 녹화본을 대기 큐에 등록한다.

#### 3) 대기 큐 기반 순차 처리
- 대기 큐에는 `queued` 상태의 녹화본만 들어간다.
- MVP 기준의 대기 큐는 외부 메시지 브로커가 아니라 DB 상태값 기반 큐를 의미한다.
- 워커는 대기 큐의 녹화본을 순차적으로 가져와 처리한다.
- 처리가 시작되면 상태를 `transcribing`으로 변경한다.
- 처리 중 오류가 발생하면 상태를 `failed`로 변경하고 종료한다.
- 모든 처리가 정상 종료되면 상태를 `completed`로 변경한다.

---

### 상세 분석 단계

녹화본 자막 생성은 한 번의 전사 호출로 끝내지 않고,  
**오디오 정리 → 청크 단위 전사 → 병합 → 자료 기반 후보정 → 목차 매핑**의 다단계 방식으로 수행한다.

---

#### 4) 오디오 추출 및 정규화
- 영상 파일에서 오디오를 추출한다.
- STT 처리에 적합한 포맷으로 변환한다.
- 필요 시 샘플레이트, 채널 수, 음량을 정규화한다.
- 무음 구간이 지나치게 길거나 손상된 구간이 있는지 점검한다.

**저장 예시**
- audio_url
- duration_seconds
- waveform_summary
- normalization_applied

---

#### 5) 언어 및 전사 설정 결정
- 사용자가 원본 언어를 지정한 경우 우선 사용한다.
- 지정되지 않은 경우, 시스템 기본값 또는 사전 감지 결과를 사용한다.
- 강의 제목, 세션 제목, 자료 키워드, 전문용어 후보를 바탕으로 STT prompt 초안을 만든다.

**설정 예시**
- source_language_code
- stt_language_code
- stt_prompt
- transcription_model

---

#### 6) 청크 분할
- 긴 녹화본은 일정 길이 또는 용량 기준으로 분할한다.
- 가능하면 문장 중간이 아니라 무음 구간 또는 발화 경계 기준으로 나눈다.
- 각 청크에는 원본 기준 시작 오프셋을 기록한다.
- 청크 사이에 약간의 overlap을 둘 수 있다.

**저장 예시**
- chunk_index
- start_ms
- end_ms
- overlap_ms
- file_url

---

#### 7) 1차 STT 수행
청크별로 1차 전사를 수행한다.

##### 기본 모드
- 모델: `gpt-4o-transcribe`
- 대안 모델: `gpt-4o-mini-transcribe`
- 응답 형식: `verbose_json`
- 타임스탬프: `segment`
- 필요 시 `word` 추가

##### 다중 화자 모드
- 모델: `gpt-4o-transcribe-diarize`
- 응답 형식: `diarized_json`

이 단계에서는 가능한 한 원문 전사에 집중하고,  
후보정은 다음 단계에서 수행한다.

**저장 예시**
- raw_transcript_text
- raw_segments
- detected_language_code
- confidence_signals

---

#### 8) 청크 병합 및 시간축 보정
- 청크별 전사 결과를 원본 녹화본 기준 시간축으로 재정렬한다.
- overlap 구간의 중복 세그먼트를 제거한다.
- 세그먼트 시작/종료 시간을 원본 영상 기준으로 보정한다.
- 병합된 전체 스크립트 초안을 생성한다.

**저장 예시**
- merged_transcript_text
- merged_segments
- segment_count

---

#### 9) 강의자료 기반 후보정
강의자료에서 이미 추출한 전역 주제, 목차, 키워드, 전문용어 후보를 이용해  
자막 텍스트를 보정한다.

보정 대상 예시:
- 과목명, 모델명, 알고리즘명 등 전문용어
- 띄어쓰기 및 문장부호
- 반복 구간 제거
- 명백한 오인식 교정

보정 원칙:
- 자막을 새로 생성하지 않고, 기존 전사 결과를 제한적으로 교정한다.
- 시간축 정보는 유지한다.
- 발화 의미를 바꾸는 과도한 재작성은 지양한다.

**저장 예시**
- refined_text
- correction_applied
- glossary_matches

---

#### 10) 자막 세그먼트 정제
- 너무 짧거나 너무 긴 세그먼트를 읽기 쉬운 단위로 재분할한다.
- 자막 한 줄 길이, 표시 시간, 가독성을 고려해 subtitle-friendly segment를 만든다.
- 필요 시 원본 STT segment와 사용자 표시용 subtitle segment를 분리 관리한다.

**저장 예시**
- subtitle_segments
- display_line_count
- reading_time_score

---

#### 11) 목차/페이지 매핑
강의자료 구조화 결과와 병합된 자막 세그먼트를 연결한다.

매핑 기준 예시:
- 목차명과 자막 키워드 유사도
- 페이지 주제문과 자막 내용 유사도
- 인접 세그먼트의 흐름 연속성
- 세션 시작 이후 시간대별 주제 전환 지점

이 단계가 완료되면,  
수강생은 자막 구간을 통해 관련 목차와 자료 페이지로 이동할 수 있다.

**저장 예시**
- section_id
- page_number
- mapping_score

---

#### 12) 자막 파일 생성
- 병합 및 보정된 세그먼트를 기준으로 자막 파일을 생성한다.
- 기본 출력은 VTT 또는 SRT 중 하나로 고정한다.
- 서비스 내부 저장용 구조화 JSON도 함께 저장할 수 있다.

**저장 예시**
- subtitle_url
- subtitle_format
- subtitle_default_language_code

---

#### 13) 검색 인덱싱 및 결과 저장
질의응답 및 탐색 기능을 위해 자막 세그먼트를 검색 가능한 형태로 저장한다.

저장 대상:
- `recordings`
  - stt_status = `completed`
  - detected_language_code
  - stt_language_code
  - subtitle_default_language_code
- `transcript_segments`
  - start_ms
  - end_ms
  - raw_text
  - refined_text
  - language_code
  - section_id
  - page_number
- 벡터 저장소
  - 자막 세그먼트 임베딩
  - 목차/페이지 연결용 텍스트

---

### 출력
- 전체 자막 스크립트
- 타임스탬프 기반 자막 세그먼트
- 자료 기반 후보정 결과
- 목차/페이지 매핑 정보
- VTT/SRT 자막 파일
- 검색 가능한 자막 임베딩 데이터

### 설계 의도

본 파이프라인은 다음 문제를 해결하기 위해 설계한다.

#### 문제 1. 긴 강의 녹화본은 한 번에 전사 처리하기 어렵다.
- 해결: 오디오를 청크 단위로 분할하고, 병합 단계에서 원본 시간축으로 다시 정렬한다.

#### 문제 2. STT 결과는 전문용어와 강의 문맥에서 오인식이 발생할 수 있다.
- 해결: 강의자료의 전역 주제, 목차, 키워드, 용어집 후보를 이용해 제한적 후보정을 수행한다.

#### 문제 3. 단순 전사 텍스트만으로는 학습 탐색 기능에 활용하기 어렵다.
- 해결: 자막 세그먼트를 목차/페이지와 매핑하여 자료, 영상, 자막이 연결된 학습 구조를 만든다.

#### 문제 4. 자막은 전사 정확도뿐 아니라 가독성도 중요하다.
- 해결: STT 원본 세그먼트와 사용자 표시용 subtitle segment를 분리해 관리할 수 있도록 설계한다.

## 운영·오류 대응 및 재시도 정책 (권장)

다음 항목들은 파이프라인을 실제 운영 환경에서 안정적으로 동작시키기 위한 권장 운영 정책입니다. MVP 단계에서는 기본값을 적용하고, 운영 환경에서 필요 시 조정하세요.

- **워커 폴링 간격:** DB 기반 큐 폴링 간격은 기본 `5초`를 권장(`WORKER_POLL_INTERVAL_SECONDS=5`).
- **재시도 정책:** 외부 API 호출(STT, LLM 등) 실패 시 최대 `3회` 재시도(`MAX_RETRIES=3`)를 권장하며, 지수 백오프(`BACKOFF_BASE_SECONDS=2`)를 적용합니다.
- **아이덴포텐시:** 업로드/처리 요청에는 `idempotency_key`를 부여해 중복 처리를 방지합니다. 워커는 `idempotency_key` 존재 시 중복 작업을 무시하거나 안전하게 이어서 처리해야 합니다.
- **상태 필드와 실패 로그:** 각 처리 엔티티(`materials`, `recordings`)에 `retry_count`, `last_error`, `last_attempted_at`, `next_retry_at` 필드를 두어 운영 진단과 재시도 스케줄링에 활용합니다.
- **임베딩 생성 시점:** `material_pages` 또는 `transcript_segments`의 처리가 `completed` 상태가 되면 워커가 임베딩 생성 작업을 트리거하여 `embeddings` 테이블(또는 벡터 저장소)에 저장합니다. 임베딩 생성 실패도 재시도 대상입니다.
- **QA 처리:** QA 요청은 동기 API로 처리합니다. API는 질의 임베딩 생성→pgvector 검색→상위 컨텍스트 결합→LLM 호출의 순으로 실행하며, 응답에 반드시 출처를 포함합니다. 동기 응답 타임아웃을 고려해 검색 결과가 충분하지 않거나 외부 호출 지연 시 명확한 에러/제한 메시지를 반환합니다.
- **모니터링 지표:** 작업 성공률, 처리 지연(queued→processing 시간), 평균 재시도 횟수, 파이프 에러 비율, 큐 적체 길이 등을 수집합니다. 기본 알림은 `재시도 초과` 및 `큐 길이 급증`일 때 발생하도록 설정하세요.
- **로그 구조:** 작업별 `job_id`, `idempotency_key`, `entity_id(materials/recordings)`, `attempt_number`, `error_type`, `external_api_code` 등을 표준화해 로그와 에러 추적이 가능하도록 합니다.

이 정책들을 문서(`docs/schema.md`, `docs/tech-stack.md`)와 코드 주석/설정 파일에 반영하면, 개발자와 운영팀이 동일한 운영 가이드라인을 따를 수 있습니다.
