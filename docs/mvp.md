# MVP 범위 정의

## 0. 공모전 전제

본 MVP는 공모전 제출용 라이브 데모를 목표로 한다.
따라서 발표용 개념 검증이 아니라,
실제 사용 가능한 웹 서비스 흐름을 최소 범위로 완성하는 것을 우선한다.

## 1. 목적

MVP의 목적은 모든 기능을 다 구현하는 것이 아니라,  
본 서비스의 핵심 가치가 가장 잘 드러나는 사용자 경험을 완성하는 것이다.

핵심 가치는 다음과 같다.

1. 강의자료를 AI로 구조화할 수 있다.
2. 녹화본을 자막과 목차 기반으로 탐색할 수 있다.
3. 수강생이 강의자료 기반 챗봇을 통해 신뢰성 있는 답변을 받을 수 있다.
4. 강의 운영과 학습 경험이 하나의 흐름으로 연결된다.

---

## 2. MVP 목표

다음 하나의 시나리오가 완전히 동작하는 것을 목표로 한다.

- 강사가 강의를 만들고 세션을 생성한다.
- PDF 강의자료를 업로드한다.
- 시스템이 PDF에서 목차와 페이지 정보를 추출한다.
- 강사가 녹화본을 업로드한다.
- 시스템이 자막을 생성하고 목차와 연결한다.
- 수강생이 자료와 영상을 학습한다.
- 수강생이 챗봇에게 질문하고 자료 근거 답변을 받는다.
- 수강생이 퀴즈를 풀고 복습한다.

## 3. MVP 포함 범위

### 3-1. 강사용
- 강의 생성
- 세션 생성
- PDF 강의자료 업로드
- 자료 처리 상태 확인

- 녹화본 업로드
- STT 자막 생성 결과 확인
- 세션 공개/비공개 설정
- 퀴즈 생성 및 공개

### 3-2. 수강생용
- 강의 목록 조회
- 강의 신청
- 수강 중인 강의 조회
- 세션 진입
- PDF 자료 열람
- 목차 클릭 기반 페이지 이동
- 녹화본 시청
- 자막 표시
- 목차 기반 영상 이동
- 챗봇 질의응답
- 퀴즈 풀이 및 결과 확인

### 3-3. AI 기능
- PDF 텍스트 추출
- 목차 추출
- 페이지별 주제문 및 키워드 생성
- STT 자막 생성
- 강의자료 기반 자막 후보정
- 자막-목차 매핑
- 강의자료 기반 QA
- 목차별 퀴즈 자동 생성

## 4. MVP 제외 범위

다음 기능은 초기 MVP에서 제외한다.

- 회원가입/로그인
- 과제 제출 및 자동 채점
- 운영자 전용 관리 기능
- 다중 강사/조교 권한 관리
- 수강생 상세 행동 로그 분석 대시보드
- 추천 학습 경로 자동 생성
- 모바일 앱 전용 기능
- 라이브 강의 기능
- 실시간 채팅/토론 기능
- 결제 기능
- 고도화된 알림 시스템

---

## 5. MVP 성공 기준

다음 기준을 만족하면 MVP가 성공적으로 구현된 것으로 본다.

1. 강사가 PDF와 녹화본을 업로드할 수 있다.
2. 자료의 목차와 페이지 구조가 생성된다.
3. 녹화본 자막이 생성되고 일부 목차와 연결된다.
4. 수강생이 목차를 클릭해 자료/영상의 원하는 위치로 이동할 수 있다.
5. 챗봇이 강의자료 기반 답변과 출처를 제공한다.
6. 퀴즈가 자동 생성되며 수강생이 풀이할 수 있다.
7. 배포 환경에서 핵심 시나리오가 실제로 시연 가능해야 한다.
8. public GitHub 저장소 구조와 문서화가 제출 가능한 수준이어야 한다.

## 6. MVP 데모 시나리오

### 데모 흐름
1. 강사 선택
2. 강의 생성
3. 세션 생성
4. PDF 업로드
5. 목차/페이지 정보 생성 결과 확인
6. 녹화본 업로드
7. 자막/타임스탬프 확인
8. 수강생 선택
9. 강의 신청
10. 세션 진입
11. 자료 열람 및 목차 이동
12. 영상 시청 및 자막 확인
13. 챗봇 질문
14. 퀴즈 풀이

### 데모 메시지
 - 단순 파일 업로드가 아니라 AI가 학습 가능한 구조로 변환한다.
 - 영상, 자료, 질문, 퀴즈가 하나의 학습 흐름으로 연결된다.

## 7. 구현·운영 결정 (요약)

- **QA 처리 방식:** 동기 API 응답을 기본으로 하며, 질문 수신 시 즉시 검색(임베딩 기반) 및 LLM 호출로 답변을 생성한다. 답변에는 반드시 출처(page/section/timestamp)를 포함한다. 자료 범위를 벗어나는 질문에는 범위 초과 안내를 명시한다.
- **비동기 파이프라인 상태:** 업로드 시 자료/녹화본 레코드가 생성되고 상태는 `uploaded` → `queued` → `processing` → `completed`／`failed` 로 전이한다. 워커는 DB 폴링 방식으로 작업을 가져가며 기본 폴링 간격은 5초로 설정한다.
- **퀴즈 공개 정책:** 퀴즈는 생성 시 기본 `draft` 상태로 저장되며, 강사가 검토 후 `published`로 전환하면 수강생에게 공개된다.
- **데모 접근 방식:** MVP 데모는 무인증 데모 모드로 제공한다. 샘플 강의·세션·자료를 기본 제공하고, 개발/데모 환경에서는 URL 쿼리나 세션 파라미터로 역할(`role=teacher|student`) 전환을 허용한다.
- **인증 관련:** 정식 회원가입/로그인은 MVP 범위에서 제외한다(문서 [4. MVP 제외 범위]와 일치).
- **재시도 및 실패 정책:** 외부 API 실패(예: STT/LLM) 시 재시도 최대 3회, 지수 백오프를 적용하고 최종 실패는 `failed`로 기록하여 UI에 재시도 안내를 노출한다.
- **임베딩/모델 선택:** 초기 MVP는 비용 우선의 임베딩/모델 설정을 사용하되, 운영 이전에 성능·비용 재평가를 수행한다.

위 결정 사항은 구현·테스트·데모 준비 단계에서 우선 적용한다. 필요 시 기술적 근거에 따라 별도 RFC를 통해 모델·서비스 변경을 진행한다.
- 교육 환경에 맞는 신뢰성 있는 AI 활용 사례를 제시한다.
